# Power Measurement Run — 2026-04-19

Multi-day battery-only run to derive average current draw from Vbat delta.
Single ADC reading has ~±80 mV noise, so we need a voltage swing well above
that floor to compute a meaningful number. On a 3000 mAh LiPo at the
expected ~13 mA component budget, 3-4 days gives ~75-150 mV.

## Firmware under test

- **Commit at flash:** `2518505` (flashed 2026-04-19 ~12:05 local)
- **Key active features affecting current:** full-rx Thread MTD (SED rejected,
  §20), SX1262 continuous RX with `setRxBoostedGainMode(true)`, MQTT
  keepalive 300 s, breathing LED PWM on P1.03, TFT backlight 30 s auto-off.
- **NOT yet live:** SX1262 `startReceiveDutyCycle()` (Phase 4 §2 — main
  future power lever, would cut ~4-5 mA).

## Battery

- **3000 mAh LiPo** (user-stated).
- **Charging history:** fully charged from USB, device unplugged from USB
  at ~12:14 local (post-flash + FATFS-config edit). Any future re-flash
  invalidates this run — re-flashing means USB which means charging too.

## Target

Come back in **3-4 days** (2026-04-22 or 2026-04-23), grab another
8-sample average, and compute:

```
avg_mA ≈ ΔV_mV / ΔV_per_mAh_on_plateau  (around 0.4 mV/mAh near 4.0 V)
       ≈ ΔV_mV × 2.5 / Δt_hours
```

Example: if end average is 3950 mV after 96 h:
- ΔV = 126 mV
- charge drawn ≈ 126 × 2.5 = 315 mAh
- average current = 315 / 96 = **~3.3 mA**

(Don't take that example as expected — it's illustrative. Actual LiPo
plateau slope varies with cell age and the flat region between 4.0 V and
3.95 V is notoriously unreliable for SoC estimation, so the result is
still ±30 %.)

## Invalidation conditions — restart the run if ANY occur

- Any `./flash.sh` operation (reflash = USB = charging)
- Device physically plugged into USB for any reason (even 5 min charges)
- Unexpected cold reboots beyond a handful (each burns a ~30 s radio init
  at elevated current; too many skews the average)
- Battery swap / external power source connected
- Thread / iot_log silence gap > 12 h with no STATUS lines captured —
  means the end-of-run Vbat can't be compared to start cleanly

## What to grep at the end

```
grep "STATUS:" /tmp/iot_e2e.log | sed 's/\x1b\[[0-9;]*m//g' | \
  awk '/<YYYY-MM-DD HH:MM range>/ { extract vbat }'
```

Plus check for reboots (uptime rewinds) and iot_log silence gaps in the
interval, both of which affect the integration.

## Periodic status checks

User will ask for status along the way. Compact report format:

```
Run age       : N.N h since 14:00 baseline
Vbat latest   : XXXX mV (baseline 4076 mV, trend ↓/↔/↑)
Vbat mean-5   : XXXX mV (avg of last 5 STATUS samples)
lora_rx total : N  (delta +M since last check)
Faults/WARN   : none | <list>
Reboots       : 0 | uptime rewound at HH:MM
iot_log gap   : 0 | N min
```

`Vbat mean-5` is the only entry worth reading for power — and only once
the delta exceeds the ±80 mV ADC noise band. The other fields catch
anomalies that would invalidate the run:

- **Reboots** — each burns a ~30 s radio init at elevated current.
  A handful over 4 days is acceptable; 10+ skews the average.
- **iot_log gaps** — if long enough to miss multiple STATUS lines, we
  lose Vbat samples in the middle and can't integrate cleanly.
- **Faults/WARN/ERR** — disconnect storms, parent-lost, anything that
  means the device isn't in steady-state operation.
- **lora_rx stuck at a fixed value** for many hours — radio isn't
  working, so the power budget no longer reflects normal operation
  (RX is the biggest continuous load in our system).

## Known-but-harmless: uptime rollover

`k_uptime_get_32()` based counters in `main.cpp` (`uptime_s`, `conn_s`,
STATUS `up=` field) roll over at **~49.7 days**. Our run is 3-4 days, so
irrelevant. A separate bug — `get_unix_epoch()` in `main.cpp:1803` would
return a past timestamp after rollover — is tracked outside this run and
unaffected over our window.
